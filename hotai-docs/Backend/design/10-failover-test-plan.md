# 故障切换测试计划与方案设计

## 1. 测试目标

验证 HotAI 在以下场景下的表现：
1. 单渠道失败时能否自动切换到备用渠道
2. 限流时能否选择其他未限流的渠道
3. 超时时是否按预期降级
4. 自动禁用的渠道恢复后能否重新启用
5. 路由决策是否符合预期（不选已失败的渠道）

## 2. 测试环境

### 2.1 前置条件

| 条件 | 要求 |
|------|------|
| 部署方式 | 本地 `go run` 或 Docker Compose |
| 渠道配置 | 同一模型（如 `gpt-3.5-turbo`）配置 ≥ 3 个渠道：A(高优先), B(中优先), C(低优先) |
| 数据库 | SQLite（开发测试） |
| 日志 | 开启 Debug 日志级别 |
| Retry | 设置 `RetryTimes = 2` |
| AutoBan | 启用 |
| 监控 | 确保 perf_metrics 正常采集 |

### 2.2 渠道配置示例

| 渠道 | 名称 | Priority | Weight | 上游 |
|------|------|----------|--------|------|
| A | OpenAI-直连 | 100 | 10 | api.openai.com |
| B | Azure-备用 | 50 | 5 | myazure.openai.azure.com |
| C | DeepSeek-兜底 | 10 | 1 | api.deepseek.com |

## 3. 测试用例

### TC-01: 单渠道失败 → 自动降级

| 字段 | 值 |
|------|-----|
| **场景** | 渠道 A 返回 500，应自动切换到 B |
| **触发方式** | 向渠道 A 的上游发送无效请求（如修改 API Key） |
| **预期** | ① 请求成功（使用渠道 B） ② 日志显示 `retry: A(500) -> B(200)` ③ 渠道 A 状态不变（500 不会触发自动禁用） |
| **验证点** | `GET /api/log/` 查看重试记录，确认渠道 B 被使用 |

### TC-02: 鉴权失败 → 自动禁用 + 降级

| 字段 | 值 |
|------|-----|
| **场景** | 渠道 A 返回 401（API Key 无效），应自动禁用并切换到 B |
| **触发方式** | 将渠道 A 的 Key 改为无效值 |
| **预期** | ① 请求成功（使用渠道 B） ② 渠道 A 状态变为 `auto_disabled`(3) ③ 管理员收到禁用通知 ④ 后续请求不再选择渠道 A |
| **验证点** | 查看渠道管理页面，渠道 A 状态为 "已自动禁用"；查看通知记录 |

### TC-03: 限流 (429) → 重试其他渠道

| 字段 | 值 |
|------|-----|
| **场景** | 渠道 A 返回 429 Rate Limit，应重试渠道 B |
| **触发方式** | 对渠道 A 所在的上游快速发送大量请求触发限流 |
| **预期** | ① 请求最终成功（使用渠道 B 或 C） ② 日志显示 `retry: A(429) -> B(200)` ③ 渠道 A **不会**被自动禁用 |
| **验证点** | 确认 429 不会触发 disable，请求最终成功 |

### TC-04: 超时 (504) → 跳过 → 不重试

| 字段 | 值 |
|------|-----|
| **场景** | 渠道 A 超时返回 504，应跳过（不重试） |
| **触发方式** | 配置渠道 A 的超短超时时间（如 1ms），或上游直接返回 504 |
| **预期** | ① 请求直接失败（返回 504） ② `shouldRetry()` 判断 `StatusCode=504` → `AlwaysSkip` ③ **没有**重试尝试 |
| **验证点** | 确认日志没有重试记录，仅一次请求就结束 |

### TC-05: 所有渠道失败 → 返回 503

| 字段 | 值 |
|------|-----|
| **场景** | A、B、C 全部不可用 |
| **触发方式** | 同时禁用或失效所有三个渠道 |
| **预期** | ① 请求最终返回 503 ② 错误消息 `no available channel` ③ 渠道 A 被禁用（第一个返回错误的）④ B 和 C 保持原状态（请求在重试失败后终止） |
| **验证点** | 检查 503 响应；日志确认所有渠道尝试均失败 |

### TC-06: 恢复后自动启用

| 字段 | 值 |
|------|-----|
| **场景** | 渠道 A 被禁用后，修复上游后自动恢复 |
| **触发方式** | ① 先执行 TC-02 禁用 A ② 修复 Key ③ 等待定时渠道测试 |
| **预期** | ① 定时测试执行时检测到 A 恢复 ② 渠道 A 状态自动变为 `enabled`(1) ③ 管理员收到启用通知 ④ 后续请求可以再次选择 A |
| **验证点** | 测试后检查渠道状态；查看通知记录 |

### TC-07: 跨组降级 (Auto Group)

| 字段 | 值 |
|------|-----|
| **场景** | Token 设置 group 为 `auto`，用户属于 [vip, default] 两个组 |
| **触发方式** | ① 配置 vip 组的所有渠道不可用 ② 使用 auto group 的 Token 请求 |
| **预期** | ① vip 组无可用渠道 → 自动切换到 default 组 ② 请求使用 default 组的渠道成功 ③ 日志显示跨组切换过程 |
| **验证点** | 查看请求日志确认最终使用 default 组的渠道 |

## 4. 测试步骤（手动执行）

### 4.1 准备

```bash
# 1. 启动服务
go run main.go

# 2. 确保 RetryTimes = 2
# 后台 → 设置 → 系统设置 → RetryTimes = 2

# 3. 确保渠道测试启用
# 后台 → 设置 → 监控设置 → 渠道测试模式 = passive_recovery
```

### 4.2 执行

对每个 TC:
1. 记录测试开始时间
2. 触发故障
3. 通过 `/v1/chat/completions` 发送请求
4. 观察响应状态码和消息
5. 检查 `GET /api/log/` 查看重试记录和渠道信息
6. 检查渠道状态（管理页面）
7. 记录测试结果到表格

### 4.4 最小验收证据

本文件是测试计划，真正提交验收时需要追加至少一次实际执行记录。建议优先执行 TC-02（鉴权失败自动禁用 + 降级）或 TC-03（429 限流重试），因为这两类最能证明平台具备故障切换能力。

每次实际记录至少包含：

| 证据项 | 要求 |
|--------|------|
| 请求证据 | 请求时间、请求模型、Token 分组、响应状态码 |
| 路由证据 | 首选渠道、重试后渠道、最终渠道 ID/名称 |
| 日志证据 | `/api/log/` 中对应 request_id 或时间范围截图/导出 |
| 状态证据 | 渠道状态变化，例如 enabled → auto_disabled |
| 结论 | 与预期一致/部分一致/不一致，并写明下一步修正 |

## 6. 改进建议（基于测试结果填写）

测试完成后根据实际表现填写以下评估：

| 维度 | 表现 | 改进建议 |
|------|------|----------|
| 重试决策准确性 | 500/429 按预期重试，504 正确跳过，401 fail-fast | - |
| 熔断及时性 | 401 触发自动禁用（需开启 AutomaticDisableChannelEnabled） | 建议文档化默认开关状态 |
| 通知有效性 | 自动禁用通知依赖 root user 配置，测试环境可能超出发送限制 | 生产环境需配置 SMTP/Webhook |
| 恢复能力 | 依赖定时渠道测试 runner，测试环境可能未启动 | CI 中可跳过或单独跑 |
| 用户体验 | 429/500 返回上游原始错误码，401 直接透传 | 可考虑统一错误消息格式 |

## 7. 实际执行记录（2026-07-16）

> 执行方式：集成测试 `tests/integration/failover_e2e_test.go`，使用 `httptest.NewServer` 模拟上游，内存 SQLite 数据库，无真实渠道/运营商依赖。

| 用例 | 执行结果 | 实际行为 | 与预期差异 |
|------|----------|----------|------------|
| TC-01 | ✅ PASS | 500 → retry → channel B(200) | 一致 |
| TC-02 | ✅ PASS | 401 → fail-fast(401)，channel A auto_disabled | 测试计划写 A→B(200)，但 Phase 4 实现为 401 fail-fast，不重试 |
| TC-03 | ✅ PASS | 429 → retry(delay≥10ms) → channel B(200) | 一致 |
| TC-04 | ✅ PASS | 504 → no retry → 504 | 一致 |
| TC-05 | ✅ PASS | 全部渠道返回 500 → 最终返回 500 | 测试计划写 503，实际返回最后渠道的错误码；仅当无可用渠道时才返回 503 |
| TC-06 | ⏭️ SKIP | 依赖系统渠道测试 runner，CI 环境未启动 | 本地手动验证通过 |
| TC-07 | ✅ PASS | auto group → default group → 200 | 一致 |

### 执行命令

```bash
go test ./tests/integration/ -run "TestFailover" -v -count=1 -timeout=120s
```

### 关键发现

1. **Phase 4 401 fail-fast 行为**：401/403 不再进入通用重试路径，直接返回上游错误码并触发自动禁用（需 `AutomaticDisableChannelEnabled=true`）。这与旧版测试计划的"降级到备用渠道"预期不同，但符合 Phase 4 设计文档。
2. **TC-05 503 vs 500**：当所有渠道均存在但全部失败时，返回最后一个渠道的错误码（500）；仅当 `GetRandomSatisfiedChannel` 返回 nil（无可用渠道）时才返回 503。
3. **RetryTimes 默认值为 0**：需在测试中显式设置 `common.RetryTimes = N` 才能触发重试。
4. **429 重试间隔**：由 `retry_setting.ratelimit_retry_interval` 控制，Phase 4 实现为 context-aware delay，不会在客户端断开后堆积 goroutine。
