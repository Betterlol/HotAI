## 一、岗位定位

> 负责让平台不只是能调用模型，而是能**稳定调用、可观测、可降级、可按策略选择更合适的渠道**。

**一句话总结：你是平台的"神经系统"——让平台看得清健康状况、做得出智能决策、扛得住异常故障。**

**适合画像：** 后端基础较好，愿意研究系统架构、日志、性能、监控、故障切换和成本优化。

### 🔹 模块二：智能路由策略设计（第 1 周完成）

**目标：** 当用户调用某个模型时，平台能根据策略"聪明地"选择最优渠道，而不是随便选一个。

**交付物：** 《智能路由策略设计文档》

**必须包含的路由策略（至少 4 种）：**

| 策略名称 | 策略逻辑 | 适用场景 |
|----------|----------|----------|
| **健康优先** | 只选择状态为健康的渠道，避开故障渠道 | 任何时候都应作为基础过滤条件 |
| **低延迟优先** | 在健康渠道中，选择历史 P95 延迟最低的 | 对响应速度要求高的实时对话场景 |
| **低成本优先** | 在健康渠道中，选择单次调用成本最低的 | 批量处理、成本敏感场景 |
| **稳定性优先** | 选择历史成功率最高、错误率最低的 | 生产环境、重要业务调用 |
| **加权轮询** | 按分数/权重分配流量，健康分高者获更多流量 | 需要均衡负载、逐步验证新渠道 |

**必须定义的故障切换规则：**

| 异常类型 | 处理动作 |
|----------|----------|
| 连续失败（≥3 次） | 标记为不健康，自动切换到备用渠道 |
| 超时（超过设定阈值） | 重试 1 次，仍失败则切换 |
| 限流（429） | 等待后重试，或切换到其他渠道 |
| 余额不足 | 立即切换到备用渠道，发送告警 |
| 鉴权失败（401） | 停止使用该渠道，发送告警，人工介入 |

### 🔹 模块三：监控指标与告警规则（第 1-2 周完成）

**目标：** 让平台"可观测"——能实时知道它在干什么、状态怎么样。

**交付物：** 《监控指标与告警规则清单》

**必须覆盖的监控维度：**

| 维度 | 具体指标 |
|------|----------|
| **流量** | 每分钟请求数（QPS）、并发连接数 |
| **成功率** | 总体成功率、各模型成功率、各渠道成功率 |
| **错误率** | 4xx 错误率、5xx 错误率、超时率、限流率 |
| **延迟** | 平均延迟、P50/P90/P95/P99 延迟、首 token 延迟 |
| **成本** | 当日/当周 Token 消耗、预估费用、各模型成本占比 |
| **渠道健康** | 各渠道在线状态、最近 5 分钟失败数、余额状态 |
| **用户** | 活跃用户数、新增 Token 数、各用户调用量 |

**告警规则示例（至少设计 5 条）：**

| 告警名称 | 触发条件 | 严重级别 |
|----------|----------|----------|
| 平台整体成功率下降 | 5 分钟内成功率 < 95% | P1（紧急） |
| 某模型全部渠道不可用 | 所有渠道连续失败 3 次 | P1（紧急） |
| 单渠道超时率过高 | 5 分钟内超时率 > 20% | P2（重要） |
| 单日成本超预算 | 当日成本超过预警线（如 100 元） | P2（重要） |
| 某渠道余额不足 | 余额 < 设定阈值 | P2（重要） |

### 🔹 模块四：故障切换测试（第 1 周完成）

**目标：** 证明平台在渠道出问题的时候，能自动切换到备用渠道，用户无感知或只感知到轻微延迟。

**交付物：** 一次故障切换测试报告

**测试场景（至少完成 3 种）：**

| 测试场景 | 操作方式 | 预期结果 |
|----------|----------|----------|
| 主动渠道下线 | 在 New API 后台禁用某个渠道 | 请求自动路由到其他健康渠道，返回成功 |
| 模拟超时 | 在测试渠道中注入延迟（如 10s+） | 触发超时重试，自动切换，日志记录 |
| 模拟返回 429 | 模拟限流响应 | 触发限流处理逻辑，切换或等待后重试 |
| 模拟鉴权失败 | 使用错误的 API Key | 渠道标记异常，告警发出，切换备用 |
| 全部渠道失败 | 模拟所有渠道同时不可用 | 返回友好错误信息，不暴露内部细节 |

**测试报告必须包含：**
- 测试时间、测试人员
- 每个场景的预期结果 vs 实际结果（截图/日志）
- 切换耗时（从故障发生到恢复的时间）
- 改进建议

## 十、常见问题 Q&A（针对你可能的疑问）

**Q1：我对 New API 不熟，怎么快速了解它的监控和路由能力？**
A：先花 2 小时阅读 New API 的官方文档，重点关注：渠道配置、模型映射、日志设置、任务系统。同时可以问 A 要已部署环境的访问权限，直接进后台看有哪些配置项。

**Q2：故障切换测试需要真实的模型 Key 吗？**
A：需要测试 Key（不是生产 Key）。你可以和 B 同学协调，用他已接入的测试模型进行故障模拟。

**Q3：架构图需要画到什么程度？**
A：第一版不用完美，但要清晰。至少让不懂技术的人也能看懂"请求从哪来到哪去"。后续可以持续迭代。

**Q4：路由策略是只在文档里设计，还是需要实现代码？**
A：第一周先在文档里设计清楚。如果 New API 已经支持配置化路由，就配置实现；如果需要二开，放入 Backlog 短期项，后续推进。
