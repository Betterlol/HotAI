## 二、项目背景与实习目标

**项目背景：** 当前岗位将拆分为"海外 AI 模型 API 聚合平台"和"国内 AI 模型 API 聚合平台"两个方向，但本次实习主线优先聚焦国内平台。

**核心目标：** 基于 New API 开源平台完成国内 AI 模型 API 聚合平台的搭建、功能配置、UI 定制、后端优化、智能监控、智能路由、模型测试、模型上架以及运营推广闭环。

### 项目最终产出
- 一个可访问的国内 AI 模型 API 聚合平台测试环境。
- 一批已接入、已测试、可调用的国内模型 API。
- 一套模型上架、测试、监控、路由和故障排查流程。
- 一版符合 HotAI 品牌风格的前端界面与开发者体验。
- 一套面向试用用户的 Quick Start、FAQ、模型介绍页和推广材料。
- 一组可复制到海外平台的文档、脚本、配置模板和运营方法。

### 实习工作原则

| 原则 | 说明 |
|------|------|
| 真实可用 | 所有任务围绕真实平台建设，不做仅停留在 PPT 或口头汇报的工作。 |
| 交付导向 | 每天必须提交可检查产出，包括代码、文档、脚本、表格、截图、Demo 或链接。 |
| 先跑通再优化 | 第一周优先完成 MVP，先让平台能访问、模型能调用、用户能试用。 |
| 文档同步 | 每个技术动作都要沉淀成部署、配置、测试或排障文档。 |
| 安全优先 | 密钥、账号、用户数据和日志必须按安全要求处理。 |

## 五、学生 A｜平台搭建与 DevOps 负责人

> **岗位定位**：负责把国内 AI 模型 API 聚合平台先跑起来，并保证其他同学有稳定环境可以接入、测试、开发和演示。

**适合画像：** Linux、Docker、Nginx、服务器基础较好，做事细致，愿意写部署和排障文档。

### 核心职责
- 完成国内平台本地与测试环境部署，优先使用 Docker / Docker Compose 方式跑通。
- 负责服务器、数据库、缓存、反向代理、域名、HTTPS、日志和备份方案。
- 建立 local / staging / prod 三类环境的配置约定，先落地 staging。
- 为其他同学提供稳定测试环境、配置说明和排障支持。
- 沉淀平台部署、升级、重启、回滚和常见故障排查文档。

### 任务目标
- 使用 New API 跑通一个 staging 测试环境，记录完整安装步骤。
- 整理服务器目录结构、端口、数据库、缓存、环境变量和管理员账号初始化方式。
- 输出 .env.example 模板，不包含任何真实密钥。
- 配置或规划 Nginx 反向代理、HTTPS、测试域名和访问入口。
- 写出第一版备份、重启、回滚和常见故障排查清单。

### 本周交付物
- 一个可访问的国内平台 staging 地址。
- 《HotAI 国内 API 平台部署手册》。
- 《环境变量与配置说明》。
- 《上线前检查清单》。
- 数据库和配置文件的最小可恢复方案。

### 验收标准
- 负责人可以按文档在新环境中复现部署流程。
- 平台能稳定访问，管理员能登录，基础页面能打开。
- B 可以在该环境中完成至少 1 个模型接入测试。
- 服务器敏感信息没有出现在代码仓库或公开文档。

### 协作接口
- 向 B 提供模型接入所需的渠道配置入口和测试环境。
- 向 C 提供前端构建、部署、静态资源替换和发布说明。
- 和 D 确认日志、监控、数据库、缓存与故障恢复方案。
- 向 E 提供用户入口、注册流程和截图素材。

## 六、学生 B｜模型接入、测试与上架负责人

> **岗位定位**：负责把公司已有合法授权的国内模型 API 接入平台，建立统一测试流程和模型上架标准。

**适合画像：** 熟悉或愿意学习 HTTP API、JSON、Python / Node.js，对大模型和模型评测有兴趣。

### 核心职责
- 梳理国内模型供应商、模型名称、价格、能力、上下文、鉴权方式和 API 格式。
- 完成 New API 渠道、模型映射、分组、倍率、令牌权限等模型上架配置。
- 编写统一测试脚本，覆盖普通对话、流式输出、多轮对话、长文本、JSON 输出等场景。
- 建立模型上架验收标准，明确模型什么情况下可以公开给试用用户。
- 向 D 提供测试数据，用于后续智能路由、监控和成本优化。

### 任务目标
- 建立《国内模型接入矩阵》，记录供应商、模型名、API Base、鉴权方式、价格、能力、限速和备注。
- 选择第一批 3-5 个候选模型，优先接入 1-2 个最容易跑通的模型。
- 配置渠道、模型映射、分组、倍率、令牌权限和模型显示名称。
- 编写统一测试脚本，至少支持普通对话、流式输出和错误码记录。
- 输出第一版测试结果：是否成功、延迟、失败原因、Token 消耗、成本估算。

### 本周交付物
- 《国内模型接入矩阵 v1》。
- 3-5 个已接入且可调用的国内模型。
- 统一模型测试脚本。
- 《模型上架验收标准》。
- 《第一批模型评测报告》。

### 验收标准
- 至少 3 个模型可以通过统一 OpenAI-compatible 调用方式完成基础测试。
- 测试脚本可以复现，输出格式清晰，便于后续接入更多模型。
- 每个模型都有能力说明、价格备注、限制说明和适合场景。
- 上架前知道哪些模型适合 Demo，哪些模型只适合内部测试。

### 协作接口
- 依赖 A 提供可用测试环境、配置入口和安全的 Key 管理方式。
- 向 C 提供模型名称、模型分类、标签、能力说明和前端展示字段。
- 向 D 提供模型延迟、成功率、错误率、价格和限速数据。
- 向 E 提供模型介绍页、Quick Start 和 FAQ 的技术内容。

## 八、学生 D｜后端架构优化、智能监控与智能路由负责人

> **岗位定位**：负责让平台不只是能调用模型，而是能稳定调用、可观测、可降级、可按策略选择更合适的渠道。

**适合画像：** 后端基础较好，愿意研究系统架构、日志、性能、监控、故障切换和成本优化。

### 核心职责
- 绘制国内平台架构图，说明用户请求、平台网关、数据库、缓存、日志、监控和上游模型关系。
- 设计智能路由策略，在同一模型存在多个渠道时按健康状态、延迟、成本、成功率进行选择。
- 设计异常监控指标和告警规则，覆盖请求数、成功率、错误率、P95 延迟、首 token 延迟和成本。
- 完成故障切换测试，模拟渠道失败、限流、超时和恢复后的处理流程。
- 形成后端优化 backlog，区分可配置项、短期二开项和长期架构优化项。

### 任务目标
- 画出国内平台初版架构图，覆盖用户请求、API 网关、数据库、缓存、日志、上游模型和监控。
- 设计第一版智能路由规则：健康状态优先、失败降权、低延迟优先、低成本优先、余额不足切换。
- 设计故障切换规则：连续失败、超时、限流、余额不足、鉴权失败时分别如何处理。
- 设计监控指标：请求量、成功率、错误率、平均延迟、P95 延迟、首 token 延迟、Token 消耗、成本。
- 调研 New API 已有监控、日志、渠道测试和任务能力，标注哪些可以配置实现，哪些需要二开。

### 本周交付物
- 《HotAI 国内 API 平台架构图 v1》。
- 《智能路由策略设计文档》。
- 《监控指标与告警规则清单》。
- 一次故障切换测试报告。
- 后端优化 backlog：配置项、短期二开项、长期优化项。

### 验收标准
- 负责人能根据架构图讲清楚请求从用户到上游模型再回到用户的完整链路。
- 路由策略能解释"为什么选这个渠道"，不是随机或只按默认配置。
- 至少完成一次手动故障测试，记录平台表现、日志表现和改进建议。
- 监控指标能支持后续成本、稳定性和用户体验分析。

### 协作接口
- 依赖 A 提供日志、数据库、缓存和部署结构信息。
- 依赖 B 提供模型延迟、成功率、失败原因、价格和限速数据。
- 向 C 提供用量、成本、渠道健康、告警信息的前端展示需求。
- 向 E 提供 FAQ 中的限流、失败、延迟、重试和异常解释。

## 十一、安全合规与负责人检查清单

### 必须提前讲清楚的安全边界
- 所有学生只使用测试 Key，不接触主账号、生产 Key 或不可撤销的高权限密钥。
- API Key、服务器密码、数据库密码不得写进 Git 仓库、截图、飞书公开文档或群聊。
